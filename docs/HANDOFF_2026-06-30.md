## What shipped, in order (5 PRs, all merged)

### PR #57 — De-duplication cleanup
Fixed cases where the same thing was implemented two different ways:
- Unified the industry categorizer (`normInd`/`INDUSTRIES`) — two copies had
  **silently drifted apart**, causing two dashboards to disagree. Now one
  source of truth in `index.js`, consumed via `ctx` everywhere else.
- Merged duplicate email-validation helpers (`isValidEmail`/`extractEmailDomain`
  → `emailSyntaxValid`/`emailDomain` from `email-validation.js`).
- Removed the unused legacy `/emails/send-microsoft` route (bypassed all
  send-window/quota/throttle safety; confirmed unused by the frontend).
- Added a guard so a manual reminder and the scheduled follow-up engine can't
  both email the same contact the same day.

### PR #58 — Event bus ("spherical" architecture)
- **New `events.js`**: a tiny in-process pub/sub over Node's `EventEmitter`.
  `emit()` is fire-and-forget and never throws into the caller; each
  subscriber is isolated (one failing subscriber can't break others).
  `EVENTS` catalog of canonical names — **always add new event types there**,
  never use raw strings.
- **New `subscribers.js`**: `registerSubscribers(deps)`, called once at the
  bottom of `index.js` after all work functions exist. This is where
  reactions live — e.g. `contact.invalidated` → cancel that contact's active
  follow-ups.
- Converted the main direct-call couplings to emit/subscribe: lead
  distribution, follow-up→auto-send, bounce→invalidate-contact.
- **New `migrations/005_domain_events.sql`**: `domain_events` table — every
  event is recorded here. **`GET /events/recent`** (admin/lead) shows the
  live timeline.

**Pattern to follow for new features:** if you're about to make Feature A
directly call into Feature B, consider instead having A `emit()` an event and
B `subscribers.js` `on()` it. Keeps things loosely coupled as the system grows.

### PR #59 — Emergency stop (global + per-manager)
- Global pause: `index.js` — `sendingPaused` (in-memory, mirrored to
  `app_settings.sending_paused`, restored on boot). Checked inside the single
  send loop (`processPendingEmailSends`) and in `autoSendForManager()`.
- Per-manager pause: `pausedManagers` Set, mirrored to
  `app_settings.sending_paused_managers` (JSON array).
- Endpoints: `GET/POST /admin/sending/status|pause|resume` (pass `manager_id`
  for per-manager, omit for global). **`GET /sending/my-status`** — any user
  can check their own pause state (added in the bugfix wave, see below).
- UI: red Emergency Stop card on the **Admin** page (global); **Stop
  emailing** button on each manager card on the **Assign Leads** page
  (per-manager, used by RA leads).

### PR #60 — Deliverability & reply-rate epic
The big one. Read `docs/DELIVERABILITY_PLAN.md` for full design rationale.
Everything here is additive/best-effort — missing migration 006 = features
silently no-op, nothing breaks.

- **`migrations/006_deliverability.sql`**: `suppression_list` table;
  `contacts.replied_at`/`reply_snippet`; `user_emails.warmup_start_date`/
  `auto_paused_at`. **Applied to prod by the owner.**
- **`deliverability.js`** (new, pure/dependency-free): disposable-domain list,
  role-account detection (`hr@`, `careers@` — **flagged, never blocked**,
  since they're legitimate recruiting targets), opt-out phrase detection,
  `scoreEmailContent()` spam heuristic. Unit-testable in isolation.
- **Reply detection** (`runReplySweep()` in `index.js`): reuses the existing
  bounce-sweep Graph inbox-reading machinery. A genuine inbound reply →
  `contacts.replied_at` set, job stage → `Connected`, active follow-ups
  cancelled, `contact.replied` emitted. An opt-out-phrase reply also adds the
  sender to `suppression_list` + emits `contact.unsubscribed`. Runs on the
  same ~30-min cadence as the bounce sweep (offset by a few minutes).
- **Suppression enforcement**: checked in the one send choke point — `info@`
  who replied "unsubscribe" never gets emailed again.
- **Warm-up ramp**: `effectiveDailyLimit` ramps 20/day +5/day from
  `warmup_start_date` if set on a mailbox; **opt-in per mailbox, off by
  default** (existing mailboxes are unaffected unless someone sets a date).
- **Bounce-rate auto-pause**: per-mailbox bounce% tracked daily in
  `app_settings`; trips at >5% with a minimum sample of 20 sent that day →
  `user_emails.auto_paused_at` set, mailbox treated as inactive in the send
  loop. Manual resume via `/admin/mailbox/:id/resume`.
- **New page: Deliverability** (sidebar, admin + leads) — health stats,
  per-mailbox warm-up/auto-pause status, reply-rate by template, spam
  checker.
- **New events:** `contact.replied`, `contact.unsubscribed`,
  `mailbox.autopaused`.

### PR #61 — Bugfix + dashboard rework (from live feedback)
- **Fixed a real bug**: `/emails/send-selected` and `/emails/queue-all` only
  checked the *global* pause, not per-manager — so a per-manager-paused BD
  clicking "Send all" got a fake "queued!" success (the send loop *did*
  correctly drop the emails, but the UX looked like a bypass). Now both
  endpoints check `isManagerPaused()` too and return a clear 409.
- Added `GET /sending/my-status` + a paused banner on the BD's own Email page
  (shown proactively, not just as an error after clicking send).
- Removed the manual "Run reply/bounce sweep" buttons from the Deliverability
  page (they run automatically; PDs don't act on individual runs).
- Removed the suppression-list *browsing* UI (felt "cloudy" to the PD) —
  **the backend enforcement is untouched and still runs silently.** Only the
  count remains, as the "Opted out" stat.
- Added 7/30/90-day filters (`?days=` on `/admin/deliverability` and
  `/analytics/templates`).
- Reply-rate-by-template now shows a human label ("Style 1") + a Preview
  popup of the actual sent subject/body, instead of a raw `v1` id.

## Deferred / explicitly NOT built (owner conversation)

1. **AI reply classifier (positive / negative / out-of-office / auto-reply /
   unsubscribe).** Reply detection today is **binary** — any inbound message
   from a contact that isn't an NDR is treated as a genuine reply, which:
   - stops follow-ups and moves the lead to "Connected" (could be wrong for
     an OOO auto-responder — a known false-positive risk flagged to the
     owner),
   - does not distinguish interested vs. not-interested.
   Owner's own words this session: *"You need to have AI for this, which we
   are not embedding right now."* — **explicitly deferred, not forgotten.**
   If revisited: the app already has Claude API access (see
   `ANTHROPIC_API_KEY` usage elsewhere in `index.js` for AI email
   generation/summaries) — the natural design is to classify each detected
   reply inside `runReplySweep()` before deciding to stop the sequence, and
   only auto-stop on a real human reply.
2. **SMTP-level / catch-all / true mailbox-existence verification** — needs a
   paid third-party verifier (ZeroBounce/NeverBounce-class). Today's
   verification is MX + disposable-domain only. Noted in
   `docs/DELIVERABILITY_PLAN.md` as a pluggable hook, not built.
3. **Full automated warm-up pool** (mailboxes auto-emailing/replying to each
   other to build reputation) — only the **ramp** (gradually raising the
   daily cap) shipped, not the inter-mailbox exchange.
4. **No UI yet to set `warmup_start_date`** on a mailbox — it's a DB column
   only. If warm-up is wanted for a real mailbox right now, it has to be set
   directly in Supabase (`user_emails.warmup_start_date`).
5. **Stop-emailing visibility for BD leads** — the per-manager pause control
   currently lives on the Assign Leads page (RA leads + admins). BD leads
   don't have an equivalent button yet (only the global Admin one, if they
   have admin role).

## Design conventions established this session (follow these going forward)

- **New cross-feature coupling → use the event bus**, not a direct function
  call, unless it must be synchronous/awaited in sequence (e.g. the
  sequential retry loop in `retryDeferredPendingSends` intentionally stays a
  direct call).
- **New schema fields → always additive** (`ADD COLUMN IF NOT EXISTS` / new
  tables), and **always read via best-effort queries that degrade safely
  if the column/table doesn't exist yet** — this is what let PR #60 ship
  code safely before the migration was applied. Never add a new column to
  the existing hot-path send queries.
- **New settings/flags → `app_settings` key-value**, mirrored to an in-memory
  variable, restored on boot (see `sendingPaused`/`pausedManagers` pattern).
- Every PR this session was verified the same way before shipping: `node
  --check` on changed files + a full `require('./index.js')` load test with
  `express.application.listen` stubbed out and dummy env vars (no real DB/
  network needed) — catches reference errors and bad wiring before deploy.
